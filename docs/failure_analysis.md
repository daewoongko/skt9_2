# Failure Analysis

## 1. 목적

본 문서는 Gemini API를 이용한 인지-운동 미션 생성 실험에서 발생할 수 있는 실패 유형을 정리한 것이다.

LLM 응답은 자연어로 보기에는 그럴듯해 보여도, 실제 서비스에 적용하려면 다음 요소를 확인해야 한다.

- 고령자에게 안전한가
- 사용자 건강 상태를 반영했는가
- 운동 결과와 인지 결과를 모두 사용했는가
- 난이도 조절이 적절한가
- 출력 형식이 서비스에서 사용 가능한가

따라서 본 프로젝트에서는 응답을 단순히 성공/실패로만 보지 않고, 실패 유형을 나누어 분석하였다.

## 2. 실패 유형 정의

| failure_type | 설명 |
|---|---|
| none | 특별한 문제 없음 |
| unsafe_exercise | 사용자 건강 상태나 균형 상태에 비해 위험한 운동을 제안함 |
| over_difficulty | 사용자 수준보다 난이도가 너무 높음 |
| vague_success_condition | 성공 조건이 구체적이지 않음 |
| weak_personalization | 사용자 데이터 반영이 약함 |
| output_format_error | JSON 형식을 지키지 않음 |
| too_verbose | 실제 서비스에 쓰기에는 응답이 지나치게 김 |

## 3. Prompt별 실패 유형 결과

Prompt A/B Test 결과는 다음과 같다.

| prompt_type | failure_type | count |
|---|---|---:|
| baseline_json | none | 1 |
| baseline_json | unsafe_exercise | 2 |
| baseline_json | weak_personalization | 1 |
| structured_v2 | none | 4 |

baseline prompt는 총 4개 케이스 중 3개에서 문제가 발생하였다.

반면 structured prompt는 4개 케이스 모두에서 큰 문제 없이 미션을 생성하였다.

## 4. Baseline Prompt의 주요 문제

baseline prompt는 JSON 형식을 요구했기 때문에 겉으로는 정돈된 응답을 만들 수 있었다.

하지만 안전 조건과 개인화 기준이 충분히 명시되지 않았기 때문에 다음 문제가 나타났다.

### 4.1 안전 조건 반영 부족

고령자 대상 미션에서는 운동 강도보다 안전성이 우선이다.

특히 다음 조건이 있는 사용자는 더 조심해야 한다.

- 무릎 통증 이력
- 고혈압 이력
- 균형 불안정
- 피로로 인한 운동 실패
- 느린 반응 시간

baseline prompt는 이러한 조건을 명확히 제한하지 않았기 때문에, 일부 케이스에서 운동 강도가 높거나 균형 부담이 있는 미션을 제안할 가능성이 있었다.

### 4.2 개인화 부족

baseline prompt는 사용자 데이터를 입력으로 받지만, 어떤 항목을 중요하게 봐야 하는지 알려주지 않는다.

그 결과 다음 정보가 약하게 반영될 수 있다.

- completed_count와 target_count 차이
- failed_reason
- balance_status
- cognitive accuracy
- reaction_time_sec
- health_notes

즉, 사용자별 데이터가 들어가도 모든 사용자에게 비슷한 미션을 추천할 위험이 있다.

### 4.3 난이도 조절 기준 부족

인지-운동 미션 게임에서는 사용자의 수행 결과에 따라 난이도를 조절해야 한다.

하지만 baseline prompt에는 다음 기준이 명확하지 않다.

- 언제 난이도를 올릴지
- 언제 난이도를 유지할지
- 언제 난이도를 낮출지
- 건강 이상이 있을 때 난이도를 어떻게 조정할지

이 때문에 응답마다 난이도 판단이 흔들릴 수 있다.

## 5. Structured Prompt가 개선한 부분

structured prompt는 baseline prompt의 문제를 줄이기 위해 다음 조건을 명확히 포함하였다.

| 개선 요소 | 설명 |
|---|---|
| 역할 지정 | Gemini를 고령자용 AI 미션 코치로 설정 |
| 안전 우선 | 고령자 대상이므로 안전을 최우선 조건으로 명시 |
| 건강 상태 반영 | health_condition과 health_notes를 반드시 고려 |
| 균형 상태 반영 | unstable이면 균형 부담이 큰 운동을 피하도록 지시 |
| 인지 결과 반영 | accuracy와 reaction_time에 따라 인지 난이도 조절 |
| 출력 형식 고정 | 서비스에서 사용 가능한 JSON 구조로 출력 |
| 위험 점검 포함 | risk_check를 통해 안전 여부를 확인 |

## 6. 케이스별 해석

| case_id | 사용자 조건 | baseline 위험 | structured 개선 |
|---|---|---|---|
| case_01 | 고령자 여성, 초보자, 건강 이상 없음 | 개인화가 약할 수 있음 | 쉬운 난이도와 느린 움직임 개선을 반영 |
| case_02 | 고령자 남성, 중급자, 건강 이상 없음 | 큰 문제 없음 | 우수한 수행을 근거로 난이도 소폭 상승 가능 |
| case_03 | 고령자 여성, 중급자, 무릎 통증 | 무릎 부담 운동 제안 위험 | 무릎 부담이 큰 운동을 피하도록 조정 |
| case_04 | 고령자 남성, 초보자, 고혈압 | 과격한 운동 또는 균형 부담 위험 | 저강도 운동과 안전 주의사항을 반영 |

## 7. 결과 해석

실험 결과 structured prompt는 baseline prompt보다 latency와 token 수가 증가하였다.

하지만 고령자 대상 서비스에서는 응답 속도만큼이나 안전성과 개인화가 중요하다.

| prompt_type | safety_score | personalization_score |
|---|---:|---:|
| baseline_json | 3.00 | 3.25 |
| structured_v2 | 4.75 | 4.75 |

이 결과는 structured prompt가 사용자 조건을 더 안정적으로 반영한다는 것을 보여준다.

## 8. 프로젝트 적용 시 주의할 점

실제 서비스에 적용할 때는 다음 점을 추가로 고려해야 한다.

1. LLM 응답을 그대로 실행하지 않고, 안전 규칙 기반 필터를 한 번 더 거친다.
2. 건강 이상이 있는 사용자는 운동 강도 상한선을 둔다.
3. 균형 불안정 사용자는 서서 하는 미션보다 앉아서 하는 미션을 우선한다.
4. 성공 조건은 숫자와 시간 기준으로 명확히 제시한다.
5. 응답이 너무 길면 게임 화면용 짧은 피드백으로 요약한다.
6. 의료적 판단이 필요한 경우 전문가 상담 안내를 포함한다.

## 9. 한계

이번 실패 분석은 소규모 실험에 기반한다.

한계는 다음과 같다.

- 사용자 케이스가 4개로 적다.
- 실제 고령자 데이터가 아니라 가상 데이터를 사용하였다.
- 점수 평가는 자동화된 정답 기준이 아니라 수동 기준에 가깝다.
- Gemini 응답은 실행 시점에 따라 달라질 수 있다.
- 실제 포즈 인식 모델의 오류 가능성은 반영하지 않았다.

## 10. 결론

Failure analysis를 통해 단순히 Gemini가 미션을 생성할 수 있는지만 확인한 것이 아니라, 어떤 조건에서 위험하거나 부족한 응답이 나올 수 있는지도 확인하였다.

baseline prompt는 간단하지만 안전성과 개인화 측면에서 불안정할 수 있다.

structured prompt는 token과 latency 비용이 더 들지만, 고령자 대상 인지-운동 미션 게임에서는 더 안전하고 일관된 결과를 제공한다.

따라서 최종 프로젝트에서는 structured prompt를 기본 방식으로 사용하고, 추가적으로 안전 필터와 평가 기준을 함께 적용하는 것이 적절하다.
