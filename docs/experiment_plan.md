# Experiment Plan

## 1. 실험 목적

본 실험의 목적은 Gemini API를 이용해 인지-운동 미션 게임에서 사용할 수 있는 AI 미션 코치 구조를 검증하는 것이다.

특히 다음 질문에 답하고자 한다.

1. Structured prompt는 baseline prompt보다 안전하고 개인화된 미션을 생성하는가?
2. 사용자별 운동 결과와 인지 결과가 다음 미션 추천에 반영되는가?
3. LLM을 실제 서비스에 사용할 때 latency와 token 수는 어느 정도 발생하는가?
4. 포즈 인식 결과와 인지 결과를 LLM 입력으로 연결할 수 있는가?

## 2. 실험 데이터

실험에는 4명의 가상 사용자 케이스를 사용하였다.

| case_id | 사용자 유형 | 수준 | 건강 상태 | 주요 특징 |
|---|---|---|---|---|
| case_01 | 고령자 여성 | 초보자 | 건강 이상 없음 | 움직임이 느리고 색깔 기억 정확도 보통 |
| case_02 | 고령자 남성 | 중급자 | 건강 이상 없음 | 운동과 인지 수행이 전반적으로 우수 |
| case_03 | 고령자 여성 | 중급자 | 무릎 통증 이력 있음 | 무릎 부담이 큰 운동 주의 필요 |
| case_04 | 고령자 남성 | 초보자 | 고혈압 이력 있음 | 피로와 균형 불안정 주의 필요 |

각 케이스는 다음 정보를 포함한다.

- 사용자 프로필
- 현재 난이도
- 운동 수행 결과
- 목표 운동 횟수
- 균형 상태
- 실패 원인
- 인지 과제 종류
- 인지 정확도
- 반응 시간
- 건강 관련 주의사항

## 3. 실험 1: Prompt A/B Test

첫 번째 실험에서는 두 가지 프롬프트를 비교하였다.

| prompt_type | 설명 |
|---|---|
| baseline_json | JSON 형식만 요구하는 기본 프롬프트 |
| structured_v2 | 안전 조건, 건강 상태, 난이도 조절, 출력 형식을 명확히 지정한 프롬프트 |

### 3.1 비교 목적

baseline prompt는 간단하지만, 고령자 대상 서비스에서 중요한 안전 조건을 놓칠 수 있다.

structured prompt는 더 길지만, 모델에게 다음 조건을 명확히 전달한다.

- 고령자 대상 안전성
- 건강 이상 여부 반영
- 균형 불안정 시 운동 강도 조절
- 무릎 통증 또는 고혈압 등 위험 요소 반영
- 인지 정확도와 반응 시간 반영
- JSON 출력 형식 고정

### 3.2 기록 지표

각 응답에 대해 다음 값을 기록하였다.

| 지표 | 의미 |
|---|---|
| latency_sec | 응답 생성에 걸린 시간 |
| input_tokens | 입력 프롬프트 토큰 수 |
| output_tokens | 출력 응답 토큰 수 |
| total_tokens | 전체 토큰 수 |
| mission_quality | 미션 품질 점수 |
| safety_score | 안전성 점수 |
| personalization_score | 개인화 점수 |
| failure_type | 실패 유형 |

## 4. 실험 2: AI Mission Coach Demo

두 번째 실험에서는 실제 프로젝트 적용 흐름을 가정하였다.

포즈 인식 모델과 인지 과제 모듈이 이미 결과를 냈다고 가정하고, Gemini가 다음 미션을 추천하도록 구성하였다.

### 4.1 입력

AI 미션 코치의 입력은 다음과 같다.

- 사용자 기본 정보
- 최근 미션 수행 이력
- 포즈 인식 결과
- 인지 과제 수행 결과
- 건강 이상 여부
- 실패 원인
- 균형 상태

### 4.2 출력

AI 미션 코치의 출력은 다음과 같다.

- 추천 미션 이름
- 미션 목표
- 인지 과제
- 운동 과제
- 난이도 변화
- 성공 조건
- 코치 피드백
- 개인화 이유
- 안전 주의사항
- 기대 효과

## 5. 평가 기준

LLM 출력은 정량적으로 완전히 평가하기 어렵기 때문에, 본 프로젝트에서는 작은 수동 평가 기준을 사용하였다.

| 평가 항목 | 점수 범위 | 평가 기준 |
|---|---:|---|
| mission_quality | 1~5 | 인지와 운동이 잘 결합된 미션인가 |
| safety_score | 1~5 | 고령자와 건강 상태를 안전하게 고려했는가 |
| personalization_score | 1~5 | 사용자별 수행 결과가 추천에 반영되었는가 |

## 6. 실패 유형

응답에서 문제가 발생할 경우 다음 유형으로 분류하였다.

| failure_type | 설명 |
|---|---|
| none | 특별한 문제 없음 |
| unsafe_exercise | 건강 상태나 균형 상태에 비해 위험한 운동 제안 |
| over_difficulty | 사용자 수준보다 난이도가 너무 높음 |
| vague_success_condition | 성공 조건이 모호함 |
| weak_personalization | 사용자 데이터 반영이 약함 |
| output_format_error | JSON 형식을 지키지 않음 |
| too_verbose | 실제 서비스에 쓰기에는 응답이 지나치게 김 |

## 7. 실험 결과 요약

Prompt A/B Test 결과, structured prompt는 baseline prompt보다 더 높은 안전성과 개인화 점수를 보였다.

| prompt_type | latency_sec | safety_score | personalization_score |
|---|---:|---:|---:|
| baseline_json | 10.46 | 3.00 | 3.25 |
| structured_v2 | 14.56 | 4.75 | 4.75 |

structured prompt는 입력 조건이 많기 때문에 latency와 token 수가 증가했지만, 고령자 대상 서비스에서 중요한 안전성과 개인화 측면에서 더 안정적인 결과를 보였다.

## 8. 실험 해석

실험 결과를 통해 다음을 확인하였다.

1. 단순 프롬프트도 미션을 생성할 수 있지만, 안전 조건 반영이 부족할 수 있다.
2. structured prompt는 건강 상태와 실패 원인을 더 잘 반영한다.
3. LLM 기반 미션 추천에서는 응답 품질뿐 아니라 latency와 token 수를 함께 봐야 한다.
4. 포즈 인식 결과와 인지 결과를 LLM 입력으로 연결하면 개인화된 AI 코치 역할이 가능하다.

## 9. 한계점

이번 실험은 소규모 재현이므로 다음 한계가 있다.

- 실제 사용자가 아닌 가상 케이스를 사용하였다.
- 포즈 인식 모델을 직접 학습하지 않고 결과값을 가정하였다.
- 평가 점수는 자동 평가가 아니라 수동 기준에 기반한다.
- 장기적인 운동 효과나 인지 개선 효과는 검증하지 않았다.
- Gemini API 응답은 실행 시점에 따라 조금씩 달라질 수 있다.

## 10. 결론

본 실험은 Gemini 2.5 논문의 개념을 인지-운동 미션 게임에 적용한 소규모 재현이다.

전체 모델 학습을 재현하지는 않았지만, 실제 서비스에서 중요한 다음 요소를 확인하였다.

- prompt 설계
- inference 결과 비교
- token 사용량
- latency
- 평가 기준
- 개인화 미션 생성
- 안전 조건 반영

따라서 Gemini는 인지-운동 미션 게임에서 사용자의 상태를 해석하고 다음 미션을 추천하는 AI 코치로 활용될 수 있다.
