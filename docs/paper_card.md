# Paper Card: Gemini 2.5 논문 분석

## 1. 선택한 논문

본 프로젝트에서는 AI 모델링 논문으로 Gemini 2.5 관련 논문을 선택하였다.

우리 팀의 프로젝트 주제는 **인지와 운동을 함께하는 미션 게임**이다.  
따라서 논문을 단순히 모델 학습 과정 중심으로 읽기보다, 실제 서비스에서 LLM을 어떻게 사용할 수 있는지에 초점을 맞추었다.

특히 다음 항목을 중심으로 논문을 분석하였다.

- Inference
- Prompt 설계
- Evaluation
- Token 수
- Latency
- Cost
- Multimodal 입력
- Tool use
- 개인화 응답 생성

## 2. 논문에서 중요하게 본 부분

Gemini 2.5 논문에서 우리 프로젝트와 가장 관련 있는 부분은 모델 학습 전체 과정이 아니라, 이미 학습된 대형 모델을 실제 서비스에 적용하는 방식이다.

우리 프로젝트는 새로운 LLM을 학습시키는 것이 아니라, Gemini API를 이용해 사용자의 상태를 해석하고 다음 미션을 추천하는 구조이다.

따라서 다음 질문을 중심으로 논문을 읽었다.

| 질문 | 우리 프로젝트에서의 의미 |
|---|---|
| 모델이 어떤 입력을 받아 추론하는가? | 포즈 인식 결과, 인지 결과, 건강 상태를 입력으로 사용 |
| 프롬프트를 어떻게 설계해야 하는가? | 안전 조건, 난이도 조절, 출력 JSON 형식을 명확히 지정 |
| 응답 품질은 어떻게 평가하는가? | 미션 품질, 안전성, 개인화 점수로 평가 |
| 비용과 지연시간은 어떻게 고려하는가? | input token, output token, total token, latency를 기록 |
| 멀티모달 모델은 어떻게 활용될 수 있는가? | 향후 카메라 기반 포즈 인식 결과와 연결 가능 |
| 실제 서비스에서는 어떤 출력이 필요한가? | AI 코치 피드백, 성공 조건, 안전 주의사항 생성 |

## 3. 논문 내용을 프로젝트에 적용한 방식

Gemini 2.5 논문에서 다룬 LLM 활용 방식을 우리 팀 주제에 맞게 다음과 같이 축소 적용하였다.

### 3.1 Inference

논문에서는 Gemini가 다양한 입력을 바탕으로 추론하는 능력을 강조한다.  
우리 프로젝트에서는 사용자의 운동 결과와 인지 결과를 입력으로 넣고, Gemini가 다음 미션을 추천하도록 구성하였다.

입력 예시는 다음과 같다.

- 사용자 유형
- 초보자/중급자 수준
- 건강 이상 여부
- 포즈 인식 결과
- 운동 성공 횟수
- 균형 상태
- 실패 원인
- 인지 정확도
- 반응 시간

### 3.2 Prompt Design

단순한 프롬프트는 그럴듯한 미션은 만들 수 있지만, 고령자 대상 서비스에서 중요한 안전 조건을 놓칠 수 있다.

그래서 본 프로젝트에서는 baseline prompt와 structured prompt를 비교하였다.

| Prompt | 특징 |
|---|---|
| baseline_json | JSON 형식만 요구한 기본 프롬프트 |
| structured_v2 | 안전 조건, 건강 상태, 난이도 조절 규칙, 개인화 기준을 명시한 프롬프트 |

실험 결과 structured prompt는 지연시간과 토큰 수가 증가했지만, 안전성과 개인화 점수가 더 높게 나타났다.

### 3.3 Evaluation

LLM 응답은 단순히 "좋아 보인다"로 평가하기 어렵다.  
따라서 본 프로젝트에서는 작은 평가 기준을 만들어 응답을 비교하였다.

평가 항목은 다음과 같다.

| 평가 항목 | 의미 |
|---|---|
| mission_quality | 추천 미션이 인지와 운동을 잘 결합했는가 |
| safety_score | 고령자와 건강 상태를 고려했는가 |
| personalization_score | 사용자별 결과를 반영했는가 |
| failure_type | 응답의 주요 문제 유형은 무엇인가 |

### 3.4 Latency, Token, Cost

Gemini API 응답에서는 token 사용량과 latency를 함께 기록하였다.

기록한 항목은 다음과 같다.

- latency_sec
- input_tokens
- output_tokens
- total_tokens

이를 통해 structured prompt가 더 안전하고 개인화된 결과를 만들지만, 그만큼 더 많은 입력 토큰과 시간이 필요하다는 점을 확인할 수 있었다.

## 4. 우리 프로젝트와의 연결

우리 팀의 최종 프로젝트는 사용자가 게임처럼 인지 과제와 운동 과제를 함께 수행하는 시스템이다.

이때 Gemini는 다음 역할을 맡을 수 있다.

1. 사용자의 최근 수행 결과를 해석한다.
2. 운동 수행 결과와 인지 결과를 함께 고려한다.
3. 다음 미션의 난이도를 조절한다.
4. 안전 조건을 반영한다.
5. 사용자에게 자연어 코치 피드백을 제공한다.

즉, Gemini는 게임 안에서 **AI 미션 코치** 역할을 할 수 있다.

## 5. 프로젝트에서 가져올 핵심 아이디어

논문에서 우리 프로젝트로 가져올 핵심 아이디어는 다음과 같다.

| 논문 개념 | 프로젝트 적용 |
|---|---|
| Inference | 사용자 상태를 바탕으로 다음 미션 생성 |
| Prompting | 안전하고 일관된 JSON 출력을 위한 structured prompt |
| Evaluation | 응답 품질, 안전성, 개인화 점수 측정 |
| Latency/Token | 실제 서비스 적용 가능성을 보기 위한 비용 지표 |
| Multimodal | 향후 카메라 포즈 인식 결과와 LLM 연결 |
| Tool use | 게임 데이터, 사용자 기록, 평가 모듈과 연결 가능 |

## 6. 결론

Gemini 2.5 논문은 대형 모델의 성능 자체뿐 아니라, 실제 서비스에서 모델을 어떻게 사용할지 생각하게 해준다.

우리 프로젝트에서는 Gemini를 직접 학습시키지는 않았지만, 논문의 핵심 개념을 다음과 같이 재현하였다.

- prompt 구조에 따른 출력 차이 비교
- token과 latency 측정
- 응답 품질 평가
- 포즈 인식 결과와 인지 결과를 이용한 미션 추천
- 고령자 대상 안전 조건 반영

이를 통해 Gemini가 인지-운동 미션 게임에서 개인화된 AI 코치로 활용될 수 있음을 확인하였다.
